# 교환 대시보드 탭별 기능 가이드

> 이 문서는 사이드바에 있는 각 화면(탭)이 **무엇을 보여주고, 누가 왜 봐야 하는지, 데이터가 어디서 오는지**를 정리합니다.
> 실제 코드를 하나하나 열어서 확인한 내용만 담았습니다 — "이렇게 되어야 한다"가 아니라 "지금 실제로 이렇게 동작한다"를 기준으로 썼습니다.
> 작동 안 하는 부분/미완성 부분도 숨기지 않고 그대로 적었습니다 (하단 [⚠️ 미완성·주의 페이지](#️-미완성주의-페이지) 참고).

**모든 데이터의 원천은 하나의 구글시트**입니다 (`api/_sheets.js`). 대시보드는 `[자사몰] 교환`, `[외부몰] 교환`, `[불량] 교환`, `재고관리` 4개 워크시트를 읽어옵니다. 재고관리 시트는 현재 **상품 마스터 정보만** 있고 실제 재고 수량 열은 없습니다 (아래 재고 부족 알림 항목 참고).

## 1. 종합 대시보드 📊
`src/pages/OverviewDashboard.tsx` · 사이드바 최상단 · **경영 보고용 첫 화면**

**목적**: 각 채널 페이지를 하나씩 안 봐도 10초 안에 "지금 상황이 어떤지 + 뭐가 급한지 + 부서별로 뭘 봐야 하는지"를 파악하는 화면. 딥다이브는 아래 2~4번 페이지가 담당하고, 여기는 요약·트리아지·경보 역할만 함.

**화면 구성** (위에서부터):
- **품질 급증 경보** (조건부) — 불량 시트에서 특정 상품이 직전기간 대비 2배 이상, 5건 이상 튀면 빨간 배너로 자동 노출. 평소엔 안 보임.
- **KPI 5종**: 전체 교환 접수, 자사몰/외부몰/불량 각 건수·비중, 미출고 합계
- **리스크 & 비용 지표 5종**: 출고 리드타임(중앙값), 출고 처리율, **회수 지연**(접수 7일+ 지났는데 반송장 미집화 — 재고 손실 리스크), **교환 배송비 수취 총액**, **반복교환 고객**(3회 이상)
- **추이 그래프**: 자사몰/외부몰/불량 일별 접수 추이 (그라데이션 영역 차트)
- **유형별 비교**: 도넛 차트
- **TOP 3 교환 발생 상품**, **미처리 요약(Aging)**, **부서별 현황**(생산/QC·물류·재무 한 줄 요약 + 해당 딥다이브 페이지 링크)

**누가 보면 좋은가**: 팀장/경영진 — 매일 처음 여는 화면. 물류·재무·생산팀도 하단 "부서별 현황" 카드만 봐도 자기 몫을 가져갈 수 있음.

**주의**: "회수 지연"은 자사몰+불량 시트 기준만 계산됨 (외부몰 시트에는 회수내역 열이 없음).

## 4. 불량 교환 분석 ⚠️
`src/pages/DefectiveAnalysis.tsx` · `[불량] 교환` 시트 기반

**목적**: "진짜 제품 결함"과 "운영 실수(전산오류 등)"·"비불량(변심 등)"을 분리해서 순수 품질 문제만 추적. 원래 불량 시트에는 변심·전산오류 건도 섞여 있어서, 그대로 집계하면 불량률이 부풀려짐.

**화면 구성**:
- 그룹 KPI 5종: 전체 접수 / **제품 결함** / **운영·물류 오류** / **비불량(변심 등)** / 확인필요(사유 미입력)
- 월별 그룹 스택 추이 (제품결함/운영오류/비불량/확인필요 색으로 구분)
- 사유별 상세 (8개 카테고리: 원단, 패드·접착, 봉제·부자재, 오염·이물질, 착용감, 기타 제품불량, 운영·물류 오류, 변심·착용후) — 각 카테고리 클릭하면 상위 5개 상품/옵션 드릴다운
- 제품결함 분포 — 상품 대분류(브라/팬티 등)별, 채널별
- **제품결함 상위 상품 TOP 3** (변심·전산오류는 제외한 순수 결함 기준) — 각 상품의 세부 불량유형 순위까지

**누가 보면 좋은가**: 생산/QC팀(제품 개선 우선순위), CX팀(품질 이슈 조기 파악).

## 5. 교환 미처리 관리 ⏳
`src/pages/StuckCases.tsx` · `[자사몰] 교환` 시트만 대상 · **실제 운영 도구** (읽기 전용 아님)

**목적**: 자사몰 일반교환 중 지연되고 있는 건을 찾아서, 셀을 직접 고치고 고객에게 알림톡을 보내는 화면. 대시보드에서 유일하게 "작업"이 일어나는 페이지 중 하나.

**노출 조건**: 옵션 누락 또는 미입금 상태면 3일 경과부터, 그 외에는 출고일이 완전히 비어있고 7일 이상 경과한 건만 "미처리"로 잡힘.

**화면 구성**:
- 경과일 버킷(3일/5일/7일/7일+) 카운트 카드
- 사유 필터(옵션 누락/미입금/장기 미회수/입고됨/입고안됨/알림톡 미발송)
- 검색(주문번호/상품명/수령자)
- **테이블에서 출고일·교환 후 옵션·지불방법 셀을 클릭해 직접 수정** → 저장 시 구글시트에 즉시 반영
- **알림톡 발송 버튼** — 건별/조건 전체 발송 가능. 발송 이력은 브라우저 localStorage에 저장 (다른 PC/브라우저에서는 안 보임)

**누가 보면 좋은가**: CX 담당자 — 매일 처리해야 하는 실무 화면.

**주의**: 알림톡 발송 이력이 서버가 아닌 로컬 브라우저에만 저장되므로, PC를 바꾸거나 캐시를 지우면 이력이 사라짐.

## 6. 교환 미처리 성과 📈
`src/pages/ExchangePerformance.tsx` · 알림톡 발송 성과 리포트

**목적**: 5번 화면에서 보낸 알림톡이 실제로 문제를 해결했는지(전환율), 그로 인해 얼마나 시간·인건비를 아꼈는지 보여줌.

**화면 구성**:
- 알림톡 성공 전환율, 평균 해결 속도, 누적 시간 절감, **인건비 절감액**
- 일간/주간/월간 발송건수·해결건수·전환율 추이 차트
- 사유별(옵션누락/미입금/장기미회수) 전환율 비교

**누가 보면 좋은가**: CX 팀장 — 자동화 알림톡 도입의 효과를 숫자로 보고할 때.

**주의**: 계산의 원천이 **브라우저 localStorage의 발송 이력**이라, 5번 화면과 마찬가지로 PC/브라우저가 바뀌면 데이터가 사라짐. 인건비 절감액은 "건당 5분, 시급 15,000원" 이라는 **하드코딩된 가정**으로 계산되므로 참고용 추정치.

## 7. 재고 부족 알림 🚨 (버그있음)
`src/pages/LowStockAlerts.tsx` · `재고관리` 시트 기반

**목적(설계상)**: 재고가 임계치 미만인 상품을 찾아 발주/교환 제한을 안내.

**⚠️ 실제 상태**: 코드가 `row.INVENTORY`(재고 수량 열)를 찾는데, 구글시트의 `재고관리` 워크시트에는 이 열 자체가 없습니다. 현재 열: `대표품번 RANK, PO, 대표품번, 대분류, SKU, CAFE24 CODE, ITEM, COLOR, SIZE`. 재고 수량이 없으므로 **필터링 결과가 항상 빈 배열이 되어 화면에 데이터가 하나도 안 뜹니다.** 재고 수량 데이터를 이 시트에 추가하거나 다른 소스와 연결해야 정상 동작합니다.

**누가 보면 좋은가**: (수정 전까지는) 아무도 유의미한 정보를 못 얻음. 재고 데이터 연동이 선행되어야 함.

## 10. 반품-교환 연동 🔁
`src/pages/Reconcile.tsx` · 실제 동작하는 자동화 도구 (구글시트 2개를 대조)

**목적**: 반품입고시트(`판토스_입고리스트`)와 교환접수시트(`[자사몰] 교환`)를 대조해서, 반품이 들어온 걸 확인하고 교환접수시트의 출고일·확인완료일을 자동으로 채워줌. 반품이 들어오기 전에 이미 출고된 "선교환" 건은 출고일은 안 건드리고 확인완료일만 채움.

**사용법**: **① 미리보기 실행** → 정상 처리 예정 건수/이슈 건수 확인 → **② 시트에 반영** 클릭. 이슈(주문없음/상품불일치/지불방법혼재 등)는 자동 반영 대상에서 빠지고 수동 확인 필요.

**누가 보면 좋은가**: CX 운영 담당자 — 반품 처리 시 매번 실행하는 도구.

## 11. 반품 시트 관리 🛠️
