## 2026-07-08 (2)

### 재고 부족 알림 페이지 버그 수정 — 재고관리 시트 J열 인식 안 되던 문제
- **문제**: `/low-stock-alerts` 화면이 항상 빈 화면이었음. 원인은 재고관리 시트의 재고수량이 **J열**에 있는데, J열은 헤더행(1행)에 이름이 없어서 `google-spreadsheet` 라이브러리(`row.get(header)` 방식)가 그 컬럼 자체를 인식하지 못했기 때문. 기존 코드는 존재하지도 않는 `row.INVENTORY` 키를 찾고 있었음. J열은 다른 시트에서 수식으로 연동해 채우는 열이라 헤더를 새로 넣지 않기로 결정(사용자 확인) → 코드에서 직접 처리.
- **해결**: `api/dashboard-data.js`, `server/index.js` 양쪽에 `'재고관리'!J2:J` 범위를 raw REST로 별도 조회해 `row.__rowIndex` 기준으로 매칭, `row['재고']` 필드로 붙여주는 로직 추가. `api/_sheets.js`에 `getJwt()`를 export해 재사용(중복 방지). `src/pages/LowStockAlerts.tsx`는 `row.INVENTORY` → `row['재고']`로 수정.
- **검증**: 시트 여러 지점(100/500/1000/2000/3000/4000행)을 직접 대조해 100% 일치 확인. 전체 5,041개 품목 중 실제 데이터가 있는 4,847개 중 4,825개가 재고 100개 미만(SKU×컬러×사이즈 세분화가 심해서 자연스러운 분포). 로컬 서버로 화면 렌더링까지 확인.
- **참고**: 이 사실은 메모리(`inventory-sheet-stock-column.md`)에 기록해둠 — 재고 관련 작업 시 항상 참고.
- **변경 파일**: `cx-exchange-dashboard/api/dashboard-data.js`, `cx-exchange-dashboard/api/_sheets.js`, `cx-exchange-dashboard/server/index.js`, `cx-exchange-dashboard/src/pages/LowStockAlerts.tsx`

### 반품입고시트 근본 성능 문제 해결 — E열 거대 수식 → 코드 대체 + 완료건 히스토리 이관
- **문제**: `/reconcile` 미리보기가 300초까지 늘려도 타임아웃되거나 "The service is currently unavailable" 오류. 원인 추적 결과 코드 문제가 아니라 **구글시트 자체가 근본적으로 느린 것**으로 확인.
- **근본 원인**: 반품입고시트(`판토스_입고리스트`) E열("구분")에 `ARRAYFORMULA(VLOOKUP(REGEXREPLACE(...), LET(...74,539행짜리 헬퍼시트를 매번 재구성...)))` 형태의 거대 수식이 29,028행 전체에 걸려있었음. 이 헬퍼시트(`[자동]자동화_교환시트연결`)조차 `[자사몰]/[외부몰]/[불량] 교환` 3개 시트를 IMPORTRANGE로 실시간으로 끌어오는 또 다른 수식 체인이었음. 구글시트는 API로 값 하나만 읽어도 이 계산을 다 마쳐야 응답을 주기 때문에, 관련 없는 열을 읽어도 지연/실패 발생.
- **해결 (2단계)**:
  1. **완료 건 자동 분리 구조 도입**: `판토스_입고리스트`(미처리 전용, 완료일 비어있는 건만) / `판토스_입고_히스토리`(완료 건 보관, 신규 시트) 로 분리. 같은 주문번호 그룹은 **모든 행의 완료일에 날짜(YYYY-MM-DD 포함, 접두어 텍스트 허용)가 있어야만** 히스토리로 이관. 하나라도 없으면 그룹 전체 보류 (부분완료/오배송 방지). 주문번호가 "확인필요"처럼 실제 주문번호가 아닌 경우도 완료일 기준으로만 판단(별도 블랙리스트 없음).
     - 원본 시트 기준: 28,028행 → 미처리 661행 + 히스토리 27,367행으로 정리.
     - 실행 스크립트는 1회성이라 저장하지 않음(터미널에서 직접 실행). 매일 자동화는 **당분간 보류** — 구글시트 폼/양식이 바뀔 수 있어 수동 운영이 안전하다고 판단.
  2. **E열 수식 → 코드 대체**: `server/scripts/recompute-category.mjs` 신규. `[자사몰]/[외부몰]/[불량] 교환` 시트의 P열(식별자)을 직접 읽어 매칭 테이블을 만들고, 반품입고시트 T열(입고코드번호=주문번호+상품명옵션)과 대조해 구분값을 계산. 기존 수식 결과와 99.95%+ 일치 검증(불일치 전부 원본 데이터 자체가 이미 `#N/A`/`#REF!`였던 셀). E3의 ARRAYFORMULA 삭제 후 계산값을 정적으로 기입. **원할 때 수동 실행**(`node scripts/recompute-category.mjs`, `--dry-run` 옵션으로 미리보기 가능).
- **결과**: 반품입고시트 읽기 속도 185초(실패) → **400ms대**로 정상화.
