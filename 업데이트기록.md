## 2026-07-06

### 반품입고시트 근본 성능 문제 해결 — E열 거대 수식 → 코드 대체 + 완료건 히스토리 이관
- **문제**: `/reconcile` 미리보기가 300초까지 늘려도 타임아웃되거나 "The service is currently unavailable" 오류. 원인 추적 결과 코드 문제가 아니라 **구글시트 자체가 근본적으로 느린 것**으로 확인.
- **근본 원인**: 반품입고시트(`판토스_입고리스트`) E열("구분")에 `ARRAYFORMULA(VLOOKUP(REGEXREPLACE(...), LET(...74,539행짜리 헬퍼시트를 매번 재구성...)))` 형태의 거대 수식이 29,028행 전체에 걸려있었음. 이 헬퍼시트(`[자동]자동화_교환시트연결`)조차 `[자사몰]/[외부몰]/[불량] 교환` 3개 시트를 IMPORTRANGE로 실시간으로 끌어오는 또 다른 수식 체인이었음. 구글시트는 API로 값 하나만 읽어도 이 계산을 다 마쳐야 응답을 주기 때문에, 관련 없는 열을 읽어도 지연/실패 발생.
- **해결 (2단계)**:
  1. **완료 건 자동 분리 구조 도입**: `판토스_입고리스트`(미처리 전용, 완료일 비어있는 건만) / `판토스_입고_히스토리`(완료 건 보관, 신규 시트) 로 분리. 같은 주문번호 그룹은 **모든 행의 완료일에 날짜(YYYY-MM-DD 포함, 접두어 텍스트 허용)가 있어야만** 히스토리로 이관. 하나라도 없으면 그룹 전체 보류 (부분완료/오배송 방지). 주문번호가 "확인필요"처럼 실제 주문번호가 아닌 경우도 완료일 기준으로만 판단(별도 블랙리스트 없음).
     - 원본 시트 기준: 28,028행 → 미처리 661행 + 히스토리 27,367행으로 정리.
     - 실행 스크립트는 1회성이라 저장하지 않음(터미널에서 직접 실행). 매일 자동화는 **당분간 보류** — 구글시트 폼/양식이 바뀔 수 있어 수동 운영이 안전하다고 판단.
  2. **E열 수식 → 코드 대체**: `server/scripts/recompute-category.mjs` 신규. `[자사몰]/[외부몰]/[불량] 교환` 시트의 P열(식별자)을 직접 읽어 매칭 테이블을 만들고, 반품입고시트 T열(입고코드번호=주문번호+상품명옵션)과 대조해 구분값을 계산. 기존 수식 결과와 99.95%+ 일치 검증(불일치 전부 원본 데이터 자체가 이미 `#N/A`/`#REF!`였던 셀). E3의 ARRAYFORMULA 삭제 후 계산값을 정적으로 기입. **원할 때 수동 실행**(`node scripts/recompute-category.mjs`, `--dry-run` 옵션으로 미리보기 가능).
- **결과**: 반품입고시트 읽기 속도 185초(실패) → **400ms대**로 정상화.
- **`api/reconcile.js` 단순화**: 미처리함이 항상 작게 유지되므로, 기존의 "뒤에서부터 청크 단위로 마지막 데이터 행 찾기"(`findLastDataRow`, `getSheetRowCount`) 로직을 완전히 제거하고 전체 범위를 한 번에 읽도록 변경. 타임아웃/재시도 안전장치(2026-07-03 도입분)는 유지.
- **작업 중 사고와 교훈**: 사본에서 테스트하던 중, 중간 단계 실패를 확인 안 하고 다음 단계(히스토리 비우기)를 진행시켜 17,555행이 일시적으로 사라짐. **구글시트 버전 기록으로 복구 성공** (데이터 유실 없음). 이후 모든 스크립트에 "각 단계 성공 확인 후에만 다음 단계 진행" 원칙 적용. 또한 `deleteDimension`으로 행을 삭제하면 시트 크기 자체가 줄어들어, 그 자리에 다시 쓰려면 그리드 크기(`gridProperties.rowCount`)를 먼저 늘려야 한다는 것도 확인.
- **적용 전 안전장치**: 원본 적용 전 스프레드시트 전체 수동 백업(`파일 → 사본 만들기`) 확보 후 진행.
- **변경 파일**: `cx-exchange-dashboard/api/reconcile.js`, `cx-exchange-dashboard/src/pages/Reconcile.tsx`(안내 문구 수정), `cx-exchange-dashboard/server/scripts/recompute-category.mjs`(신규), 구글시트(`판토스_입고리스트`/`판토스_입고_히스토리`/E열).

## 2026-07-03

### 반품-교환 연동(reconcile) 타임아웃/오류 해결 + 선교환 처리 추가
- **문제**: "미리보기 실행" 버튼이 `Vercel Runtime Timeout Error: Task timed out after 60 seconds`로 자주 실패. maxDuration을 늘렸더니 이번엔 구글 시트 API가 `The service is currently unavailable`(503류 일시 오류)를 반환.
- **원인**: 마지막 데이터 행을 알아내려고 반품입고시트 J열/교환접수시트 G열 **전체(수만 행)를 한 번에 읽는 요청**이 병목이자 구글 API 거부의 원인이었음.
- **조치 (3단계, 순서대로 적용)**:
  1. `vercel.json` — `api/reconcile.js` `maxDuration` 60 → 300초
  2. `api/reconcile.js` — 구글 API가 일시적 오류(503 등)를 주면 최대 3회 지수 백오프로 자동 재시도 (`withRetry` 헬퍼, `sheetsGet`/`sheetsBatchUpdate`에 적용)
  3. `api/reconcile.js` — 열 전체를 한 번에 읽던 방식을 제거. 시트 크기 상한은 메타데이터(`gridProperties.rowCount`)로 가볍게 조회하고, **뒤에서부터 1만 행씩 나눠서(`findLastDataRow`)** 데이터가 있는 마지막 행을 탐색하도록 변경. 이후 최근 7,000행만 읽는 기존 로직은 유지.
- **결과**: 미리보기 실행 정상 동작 확인 (62건 정상 처리 / 9건 이슈, 타임아웃·오류 없음).
- **선교환 건 처리 추가**: 반품이 들어오기 전에 이미 새 상품을 출고한 "선교환" 건(교환접수시트 출고일 열에 이미 날짜가 채워진 경우)을 감지해, **출고일은 건드리지 않고 반품입고시트 확인완료일만 기입**하도록 로직 추가.
  - 미리보기 화면 출고일 컬럼에 `이미출고됨(2026-07-03)` 형태로 표시, 사유 컬럼에 `2026-07-03 선출고 → 확인완료일만 기입` 표시
  - `parseExchanges`에 `existingShipDate` 필드 추가, `reconcile`에서 우선 분기 처리, `applyActions`에서 `skipShipWrite` 플래그로 해당 건은 교환접수시트 쓰기 제외
- **변경 파일**: `cx-exchange-dashboard/vercel.json`, `cx-exchange-dashboard/api/reconcile.js`
