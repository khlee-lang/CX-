## 3. 이게 무슨 도구인가 (처리 흐름)

### 데이터의 원천
모든 데이터는 **하나의 구글시트**에서 옵니다 (9번 참고). 대시보드는 이 시트를 **읽고**, 일부 셀(출고일·지불방법·교환옵션)은 화면에서 **고쳐 쓸 수도** 있습니다.

### 알림톡 발송 흐름 (핵심)
"교환 미처리 관리" 화면에서 **📨 발송** 버튼을 누르면:

```
브라우저 (cx-bice.vercel.app)
   │  발송 버튼 클릭
   ▼
Vercel 서버리스 함수 (api/send-alimtalk.js)
   │  주문 상태(입고/미입고 × 옵션누락 × 미입금)에 따라 분기
   ▼
api/_alimtalk.js  (메시지 템플릿 + 발송 함수, JS로 이식)
   │
   ▼
루나소프트(Lunasoft) 알림톡 API → 고객 휴대폰
```

- **입고 여부 × 옵션 누락 여부 × 미입금 여부** 조합으로 7가지 메시지 템플릿(50195~50201) 중 하나가 자동 선택됩니다. 분기 로직은 `api/_alimtalk.js`.
- 발송 이력(언제 보냈는지)은 **브라우저 localStorage**에 저장됩니다. → **다른 브라우저/PC에서는 안 보입니다.** (서버에 저장되는 게 아님)
- ※ 2026-06-30 이전: Python(`send_single_python.py` + `alimtalk_core.py`) 기반이었으나, Vercel 배포를 위해 JS로 완전 이식. Python 파일은 레거시로 남아 있음.

### 화면 구성 (사이드바 메뉴 = src/pages/)
개요, 자사몰 교환, 외부몰 교환, 불량 분석, 리드타임, 교환 실적, 재고 부족 알림, **교환 미처리 관리(StuckCases — 알림톡 발송이 여기 있음)**, 상품별 상세, 리포트 센터.

**[자동화 섹션 — 2026-06-30 추가, 2026-07-03 안정화, 2026-07-09/10 개선]**
- **반품-교환 연동** (`/reconcile`): 반품입고시트(`판토스_입고리스트`)와 교환접수시트(`[자사몰]`/`[외부몰] 교환`, 화면 상단 탭으로 전환)를 대조해 출고일·확인완료일을 자동 기입. ① 미리보기 → ② 시트에 반영 2단계로 동작.
- **선교환 처리**: 반품이 들어오기 전에 이미 출고된 건(교환접수시트 출고일 열에 날짜가 이미 있음)은 출고일을 건드리지 않고 반품입고시트 확인완료일만 기입. 미리보기에 `이미출고됨(날짜)`로 표시됨.
- **열 참조 방식(2026-07-09)**: 반품입고시트 열이 실시간으로 추가/삭제되는 걸 겪은 뒤, 모든 열 참조를 고정 위치가 아니라 **헤더 텍스트로 매번 찾는 방식**(`api/_columns.js`)으로 전환. 헤더를 못 찾으면 에러로 바로 드러남.
- **카테고리혼재 감지(2026-07-10)**: 같은 주문번호로 반품입고시트에 구분값이 다른 행이 섞여 들어오면(예: 자사몰교환 신청 상품 5개 + 신청 안 한 상품 5개가 같이 반품됨) 예전엔 신청한 항목 개수만 맞으면 조용히 "정상 처리"로 통과됐음. 이제 이런 경우 `카테고리혼재` 이슈로 분리해서 사람이 확인하게 함.
- **타임아웃/일시오류 대응(2026-07-03)**: 초기엔 마지막 데이터 행을 찾으려 열 전체(수만 행)를 한 번에 읽다가 60초 타임아웃 또는 구글 API 일시 오류(503)가 자주 발생. ① `maxDuration` 60→300초, ② 구글 API 일시 오류 시 자동 재시도(최대 3회), ③ 열 전체 대신 뒤에서부터 1만 행씩 나눠 읽어 마지막 데이터 행 탐색 — 세 조치로 해결. 자세한 내용은 [업데이트기록.md](업데이트기록.md)의 2026-07-03 항목 참고.

## 6. 결과 읽는 법 (정상 vs 수동 확인 필요)

### 정상
- 서버 탭: `Server running at http://localhost:3001`
- 프론트 탭: `Local: http://localhost:5173/`
- 화면: 표에 교환 건들이 보이고, 새로고침(🔄) 버튼이 돈 뒤 데이터가 채워짐.
- 알림톡 발송 성공: 버튼 아래 `✅ (시각)` 초록 배지.

### 사람이 수동으로 확인해야 하는 케이스
- **알림톡 발송 실패 → 빨간 `🔁 재발송` 버튼**: 보통 ① 전화번호가 `010`으로 시작하는 11자리가 아님, ② 루나소프트 API 거절. 정확한 원인은 `cx-exchange-dashboard/server/alimtalk.log` 를 열어 마지막 기록 확인.

## 7. 실행 옵션

- **기본:** 위 2번 방식(터미널 2탭). 가장 안정적.
- **클로드에게 부탁:** "백 프론트 실행해줘" 라고 하면 클로드가 켜줍니다. 단, 클로드 작업 세션이 끝나면 같이 꺼질 수 있어 **장시간 운영엔 부적합**.
- **포트:** 백엔드 `3001` 고정(`server/index.js:10`), 프론트 `5173`(Vite 기본). 다른 프로그램이 점유 중이면 충돌. 확인: `lsof -ti:3001`, `lsof -ti:5173`.
- **알림톡 단건 직접 테스트(개발용):** `send_single_python.py` 에 JSON 인자를 줘서 실행 가능하나, **실제 고객에게 발송되므로** 테스트는 본인 번호로만.

## 부록: 문서와 실제가 달랐던 점

조사 중 **기존 문서/코드와 실제가 다른** 부분들입니다. 다음 사람이 속지 않도록 정리합니다.

1. **"cron 자동 실행" 아님** — 이 폴더를 가리키는 crontab 항목은 **하나도 없습니다**(`crontab -l`로 확인). 대시보드와 이 폴더의 파이썬 스크립트는 **자동 실행되지 않으며, 전부 수동**입니다. (참고: 같은 맥의 crontab에는 *다른* 프로젝트들의 자동화는 등록돼 있습니다.)

2. **`대시보드_실행방법.md` 의 마지막 줄이 사실과 다름** — "제가 현재 배경에서 이 명령어들을 실행해 두었으니 바로 접속하면 된다"고 적혀 있으나, 서버는 **상시 떠 있지 않습니다.** 매번 직접 켜야 합니다 (실제로 작업 중 여러 번 꺼짐을 확인).

3. **`교환미처리건알림톡.py` 는 작동하지 않는 레거시** — `alimtalk_core`에 **존재하지 않는 함수**(`same_option`, `send_jeju_notice`, `send_exchange_notice` 등)를 호출하고, 그나마 모듈 접두사 없이 부르고 있어 실행 시 에러가 납니다. 파일명만 보면 "미처리 알림톡 자동화"처럼 보이지만 **실제 발송은 웹 대시보드 흐름**(`send_single_python.py`)이 담당합니다.

4. **`README.md` 는 프로젝트 설명이 아님** — Vite 기본 템플릿 그대로라 프로젝트 정보가 없습니다. 실제 가이드는 이 문서와 `대시보드_실행방법.md` 입니다.

5. **`.env` 파일이 없는데 `dotenv.config()` 호출** — `server/index.js`가 `.env`를 읽으려 하지만 파일이 없습니다. 다만 서버 로그에 `injecting env (0)`로 뜨며 **무해하게 넘어갑니다**(현재 환경변수로 쓰는 값이 없음). 별도 조치 불필요.
